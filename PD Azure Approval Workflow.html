The Catalyst for Change

This section outlines the primary drivers behind the need for a formalized deployment process. Recent incidents have highlighted the risks of direct production access, emphasizing the need to transition to a more secure, enterprise-level workflow.

⚠

Direct Production Pushes

Code pushed directly to production from local environments (e.g., Visual Studio) bypasses safety checks, leading to severe layout breaks (e.g., footer updates breaking the 2026 UI).

⇄

Environment Discrepancies

Manual updates to server files (like JSON warranty data) create environments that are out-of-sync with version control, threatening data overwrites on the next automated deployment.

🚧

Lack of Dev Testing

Changes are frequently made without being verified in the Development or Test environments first. The new objective is strictly: "The only place you should be testing things is development."

Resource Planning Options

To implement the "Two-Approver Gate" successfully, Switchbox requires a defined cadence for reviewing Pull Requests (PRs). Review the options below to decide how to allocate review resources.

Option A
⏰

Scheduled Cadence

A dedicated, blocked timeframe (e.g., Monday afternoon for 2 hours) exclusively reserved for PR reviews and approvals.

✔ Predictability: Deployments happen on a known schedule. Easy to budget.

✖ Bottlenecks: Code sits idle until the scheduled window. Not ideal for rapid, mid-week feature releases.

Option B
🔔

Ad-Hoc / As-Needed

Reviewers are alerted via Azure DevOps when a PR is submitted and review it as soon as bandwidth permits.

✔ Flexibility: Code can move to production on any day of the week once approved.

✖ Variable Turnaround: No guaranteed SLA for reviews. Relies on reviewers having open time during the day.

⛑ Emergency Protocols

While the goal is to eliminate manual pushes, exceptions exist for critical downtime (e.g., "Sunday Night Site Down").

  • Management can authorize an override command to deploy immediately.
  • Any manual hotfix applied to JSON or site files must be retroactively committed to Azure DevOps to maintain the Source of Truth.
  • Sitefinity template changes are mitigated by the new Manifest JSON configuration, separating static app builds from CMS templates.
